# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A Revit external command add-in (C#/.NET) named **WallSplitter**, supporting **Revit 2023–2027** from a single codebase (multi-targeted, one build per year — see "Multi-version build" below).

- `WallSplitter/App.cs` — `App : IExternalApplication`. On startup, creates a dedicated ribbon tab ("Sunny Tools") with panels "벽체 분리" (button "벽체 분리"→`SplitWallCommand`), "바닥 분리" (button "바닥 분리"→`SplitFloorCommand`, a separate panel added 2026-07-13 — deliberately its own panel, not folded into "벽체 분리", per explicit request even though it shares all naming/type-assignment settings with walls), and "NAMER" (button "NAMER"→`NamerCommand`) — instead of registering under the default Add-Ins tab's External Tools dropdown. **Both** "벽체 분리" and "바닥 분리" panels get their own "설정"→`SettingsCommand` stacked with a small "단일/복수" toggle button→`ToggleTypeAssignmentPersistenceCommand` (via the shared `App.AddSettingsStack` helper, `RibbonPanel.AddStackedItems`) — added 2026-07-13 after the wall-only placement made the (fully shared) settings feel inaccessible while working on floors. Since a `PushButtonData`'s internal name must be unique across the whole application (not just per panel), the floor panel's copies use an `idSuffix` ("`_Floor`") appended to the same internal name prefixes, both pointing at the same `SettingsCommand`/`ToggleTypeAssignmentPersistenceCommand` classes so they read/write the one shared `NamingSettings` file. The toggle button's `ItemText` ("단일"/"복수") is kept in sync after each click via a static list of `PushButton` references (`App._typeAssignmentToggleButtons`, one per panel copy) and `App.UpdateTypeAssignmentToggleLabel`, since a `PushButtonData` command has no other way to update its own already-created ribbon item. The `.addin` manifest registers **only** `Type="Application"` pointing at `WallSplitter.App` — there is no `Type="Command"` entry anywhere; all command classes are referenced solely via `PushButtonData` inside `App.OnStartup`. Button icons are `WallSplitter/Resources/icon{,_settings,_namer,_floor}{16,32}.png` (embedded resources, `LogicalName` set explicitly in the csproj, generated via `System.Drawing`), loaded via `BitmapImage` — this requires `<UseWPF>true</UseWPF>` in `WallSplitter.csproj`. `icon_floor{16,32}.png` was redrawn 2026-07-13 from a plain letter-in-circle placeholder to 3 stacked/staggered horizontal bars (a floor-slab cross-section split into layers, mirroring `icon32.png`'s vertical-fin wall-layer look but horizontal) to match the visual tier of the other main-feature icons. `OnStartup` also calls `EnsureWpfApplication()` first thing — Revit's process never creates a `System.Windows.Application`, and some WPF controls with complex default (themed) styles fail their `InitializeComponent()` with an opaque `"Provide value on 'System.Windows.Baml2006.TypeConverterMarkupExtension' threw an exception"` error without one (simple controls like `Button`/`TextBox`/`ComboBox` used in `SettingsWindow`/`NamerWindow` didn't hit this). The fix is instantiating `new System.Windows.Application { ShutdownMode = ShutdownMode.OnExplicitShutdown }` once, guarded by `Application.Current == null` — never call `.Run()` on it, Revit owns the message loop.
- `WallSplitter/NamingSettings.cs` + `SettingsWindow.xaml(.cs)` + `SettingsCommand.cs` — the "설정" button opens a WPF window (owned by the Revit main window via `WindowInteropHelper` so it can't hide behind Revit) where the user builds the single-layer `WallType` naming template as an ordered, reorderable list of tokens (`ProjectName`/`BuildingName`/`Custom` are free-text typed in the window; `Material`/`Thickness` are **not editable** there — resolved per-layer at split time), each followed by a chosen separator (`""`/`-`/`_`). Saved as JSON to `%APPDATA%\WallSplitter\naming-settings.json` (global, cross-project; save failures are caught and shown in a MessageBox, never thrown out of the WPF handler); `Load()` falls back to the default template (`단일_{재질명}_{두께mm}mm`) if the file is missing/corrupt. `BuildName` strips Revit-forbidden name characters (`\ : { } [ ] | ; < > ? ` ~` + control chars) and falls back to the default template if the result is blank. Naming logic is Revit-independent and covered by a console smoke test (scratchpad `NamingTest`, all passing as of 2026-07-09). The move-up/move-down/remove row buttons use hand-drawn `Polygon`/`Line` shapes (`CreateTriangle`/`CreateXMark`), not text glyphs (`▲`/`▼`/`✕`) — live-tested invisible, likely because this WPF window shares Revit's process and Revit's own (possibly dark) global style resources bled into it, and/or the glyphs weren't in the button font. `SettingsWindow.xaml` also pins `Window.Resources` styles forcing black text on light `Button`/`TextBox`/`ComboBox`/`TextBlock` backgrounds for the same reason — don't remove either fix without re-testing live.
- **`NamingSettings.Mode`** (added 2026-07-13): `SettingsWindow` is now a `TabControl` with two tabs, each corresponding to a `NamingMode` — selecting a tab and clicking 저장 sets `Mode` accordingly (index 0=`Template`=the token-list behavior above, index 1=`DirectType`). In `DirectType` mode, `SplitWallCommand` skips `NamingSettings.BuildName`/`GetOrCreateSingleLayerWallType` entirely: for every original wall, before Tx1 opens (modal WPF dialogs are never shown inside an open transaction, same rule as `NamerCommand`), a `LayerTypeAssignmentWindow` lets the user search/pick an existing `WallType` (`WallKind.Basic` only) per layer, and that exact type is assigned to the new wall as-is. `NamingSettings.TypeAssignmentPersistence` (`Single`/`Multiple`, the ribbon "단일/복수" toggle) controls whether that per-layer picker reappears for every original wall (`Single`) or is remembered and auto-reapplied to the next wall whose layer composition matches (`Multiple`, via `TypeAssignmentSession`, a static in-memory-only session keyed to material+width per layer index and the current document — never persisted to disk, since a `WallType` `ElementId` is meaningless outside the document it came from). On any mismatch (layer count or a layer's material/width differs from the last pick) the picker reopens with a red-text reason (`LayerTypeAssignmentWindow`'s `warningMessage`); canceling the picker cancels the whole `SplitWallCommand` run. `SplitWallCommand.BuildLayerDescriptors` (pure read, called before any transaction) computes per-layer offset/material/width once and is shared by both modes' Tx1 wall-creation loop and the `DirectType` picker/signature, replacing what used to be inline per-layer math inside Tx1.
- `WallSplitter/Class1.cs` — `SplitWallCommand`, an `IExternalCommand`. Selection: `uiDoc.Selection.GetElementIds()` is checked first — if the user already selected element(s) before clicking the button, those are used directly (filtered to `Wall`s); only if nothing is pre-selected does it fall back to interactive `PickObjects` + `WallSelectionFilter` (ESC → `OperationCanceledException` → `Result.Cancelled`, not an error). Work is split into **three phases** because `SketchEditScope` cannot `Start()` while a transaction is open, and edits inside a scope need their own inner transaction. Do NOT collapse these back into one transaction — that silently breaks profile copying.
  1. **Tx 1 ("단일 벽 유형 생성 및 벽체 분리")**: per layer — skip zero-width (membrane) layers; resolve type name via `NamingSettings.BuildName`; `GetOrCreateSingleLayerWallType` reuses an existing type only after verifying material+width match, otherwise appends `(2)`, `(3)`…; create the offset wall via `OffsetCurvePerpendicular`; copy base/top constraints; `DisallowWallJoinAtEnd` both ends; for walls with an edited profile, build a `ProfileTask.Mapping`, call `CreateProfileSketch()`, queue the task. Delete all originals, commit.
  2. **Profile phase (no outer tx)**: per `ProfileTask`, `SketchEditScope.Start` → inner `Transaction` (delete default rectangle curves, add original profile curves transformed by `task.Mapping`) → `scope.Commit(SilentFailuresPreprocessor)`.
  3. **Tx 2 ("같은 재료 벽 재결합")**: `CollectJoinPairs` matches original-wall end joins (recorded pre-delete from `get_ElementsAtJoin`, excluding self) between selected walls, paired by matching `WallType`. For each pair, `TrimToIntersection` then `AllowWallJoinAtEnd` on both.

  **Three root-cause fixes, all QA-driven by live testing (2026-07-09), not guesses — don't revert any of these without re-testing live:**
  - **`OffsetCurvePerpendicular`**: `Line` → plain vector translation (`orientation * offset`); a `Curve.CreateOffset`-based version was live-tested and piled walls at the centerline. `Arc` → **manual `Arc.Create` reconstruction** (`OffsetArcPerpendicular`): keeps the same `Center`/`GetEndParameter(0,1)`/`XDirection`/`YDirection`, only changes the radius, trying `radius+offset` then `radius-offset` and keeping whichever candidate's midpoint actually moved toward `orientation`. `Curve.CreateOffset` on `Arc` was live-tested to throw `"Cannot create the offset of the curve"` even with a non-negative distance — it is **not used at all** for arcs anymore, by design; don't reintroduce it.
  - **Profile copy** (`ProfileTask.Mapping`, applied via `TransformCurveOrFallback` which falls back to a straight line between mapped endpoints if `CreateTransformed` rejects the result — e.g. a profile `Arc` distorting into a non-circular shape under the Arc case's non-uniform scale): Line → `Transform.CreateTranslation(orientation * centerOffset)` (exact — translating a vertical plane by the same vector that translates the line within it yields the unique vertical plane through the offset line). Arc → `BuildArcProfileMapping`, a radial scale about the arc's center by `offsetArc.Radius / originalArc.Radius`, Z untouched (exact — offset arcs are concentric, so any point at the old radius maps to the corresponding point at the new radius by that exact ratio). **The geometry math alone was not sufficient**: `SilentFailuresPreprocessor` originally only called `DeleteWarning` for `FailureSeverity.Warning`, but "구속조건이 충족되지 않습니다" is `FailureSeverity.Error` and was live-tested to still show its interactive resolution dialog regardless of the preprocessor. Fixed by calling `failuresAccessor.ResolveFailure(f)` (applies the failure's own default resolution, same effect as clicking the dialog's highlighted button) for any non-Warning failure that `HasResolutions()`, and returning `ProceedWithCommit`.
  - **Corner join** (`TrimToIntersection`, Line-based wall pairs only): each layer's endpoint is `original endpoint + offset vector`, so among several layers at one corner, only the layer whose offset happens to be ~0 (coincides with the original centerline) has its endpoint still exactly matching the adjacent wall's corresponding layer — that's why only the "center" wall auto-joined and every offset layer needed manual trimming. `TrimToIntersection` computes the actual XY line-line intersection of the two (already-offset) location lines and reassigns each wall's `LocationCurve.Curve` endpoint to it *before* `AllowWallJoinAtEnd`, so Revit's native join/miter engine has genuinely coincident endpoints to work with, the same way it already does for the centerline layer.

  **No completion popup** on success (removed on request), but skipped walls / profile-copy failures do show a single "경고" dialog.

- **`WallSplitter/SplitFloorCommand.cs`** (added 2026-07-13): `SplitFloorCommand`, the same design as `SplitWallCommand` applied to `Floor`/`FloorType` — same selection pattern, and **fully shares** `NamingSettings` (`Mode`/`TypeAssignmentPersistence`, same "설정" window, same ribbon toggle) with walls. To make this sharing safe, `TypeAssignmentSession` (used by `TryGetReusableSelection`/`Remember`) now keys its remembered layer-signature/selection state by a `TypeAssignmentTarget` (`Wall`/`Floor`) enum, so a wall's and a floor's "복수" sessions never cross-contaminate even if their layer (material, width) signatures happen to coincide — without this, reusing a remembered `WallType` Id as a `FloorType` Id would throw. `LayerTypeAssignmentWindow`/`LayerPickItem` were generalized from `WallType` to `ElementType` for the same reason (one picker window now serves both commands). Floors differ from walls in ways that meaningfully simplify the port: floor layers offset **vertically** (down from the original boundary sketch's Z, assumed to be the top/finish surface) instead of horizontally from a wall's location curve, and every `Floor` always has a boundary sketch (`Floor.SketchId`) that `Floor.Create(doc, curveLoops, floorTypeId, levelId, isStructural, slopeArrow, slope)` accepts directly — so there is no wall-style "edited profile" branch, no `SketchEditScope` second phase, and (since every new layer keeps the exact same X/Y footprint, just at a different Z) no corner-trim/rejoin phase either; **one transaction is enough**. `BuildLayerDescriptors` computes each kept layer's `TopOffset` (downward depth from the original top, membrane layers skipped exactly like walls). Slab-shape-edited (sloped) floors are skipped with a reason (`SlabShapeEditor.IsEnabled`), analogous to walls skipping non-Line/Arc location curves — no attempt is made to replicate per-vertex shape edits per layer.
  - **CONFIRMED LIVE BUG (2026-07-13), fixed**: the original approach translated each boundary loop's Z via `CurveLoop.CreateViaTransform(loop, Transform.CreateTranslation(new XYZ(0,0,-TopOffset)))` before `Floor.Create`, on the assumption `Floor.Create` would honor the input curves' Z for a flat (non-sloped) floor. Live-tested false: with `slopeArrow=null`/`slope=0`, `Floor.Create` **ignores the profile curves' Z entirely** and always places the floor at the level's plane (i.e. "Height Offset From Level" = 0), so every split layer came out flush with the level regardless of its actual depth. Fixed by dropping the Z-translation (boundary loops are now passed to `Floor.Create` untouched, at the original sketch's Z, since it has no effect either way) and instead explicitly setting `BuiltInParameter.FLOOR_HEIGHTABOVELEVEL_PARAM` on the newly created `Floor` to `prep.OriginalHeightOffset - d.TopOffset` (`OriginalHeightOffset` = the source floor's own Height Offset From Level, read once per floor in `FloorPrepData`, before Tx). This also fixed the previously-undocumented `CompoundStructure.GetLayers()` layer-order/top-surface assumption implicitly, since offset direction is now visibly correct after live testing.
- **First use of year-conditional compilation** (`WallSplitter.csproj`'s `DefineConstants`, added 2026-07-13 for the floor feature): `Floor.SlabShapeEditor` is a property in the 2023 API, a `GetSlabShapeEditor()` method from 2024 on (2024/2025 expose both, deprecated property removed by 2026) — confirmed by reflecting all 5 years' `Nice3point.Revit.Api.RevitAPI` packages via `MetadataLoadContext` (scratchpad `ApiProbe`), not guessed. Gated with `#if REVIT2023 / #else` in `SplitFloorCommand.cs`, using a `REVIT2023`/`REVIT2024_OR_GREATER` constant pair defined per `$(RevitVersion)` in the csproj (Nice3point SDK's `_OR_GREATER` naming convention, hand-rolled since we don't use their SDK — see "Multi-version build" below). This is the first time this pattern was needed since the project's start; add more such constants the same way if another API is found to differ by year.

UI strings and code comments are in Korean; keep new user-facing strings and comments in Korean to stay consistent with the existing file.

**Known limitations** (QA-reviewed 2026-07-09, accepted for now): doors/windows hosted on an original wall are deleted with it (not re-hosted onto the new layer walls); `TrimToIntersection` only handles Line-based wall pairs (Arc corners still rely on bare `AllowWallJoinAtEnd`, untested); only end-joins between *selected* walls are restored — T-joins/`JoinGeometryUtils` geometry joins are not; thickness display rounds to whole mm so 12.3mm vs 12.4mm layers merge into one type name; new walls are always non-structural and default room-bounding; stacked/vertically-compound wall types are skipped or untested; net48 payload's `System.Text.Json` can theoretically version-conflict with other net48 add-ins in Revit 2023/2024.

## NAMER (이름 일괄 변경)

A batch-rename tool for Views/Sheets/Families/Types/Materials, added 2026-07-10. `NamerWindow` is **modal** (`ShowDialog()`), not modeless — a one-shot batch operation doesn't need live two-way sync, so `NamerCommand.Execute` can just run a `Transaction` directly after the dialog returns (still inside the valid API context, since `ShowDialog()` blocks synchronously within `Execute`) with **no `ExternalEvent` needed at all**. This is deliberately the simpler architecture; don't "upgrade" it to a modeless pattern unless a real requirement (e.g. live preview against concurrent Revit edits) demands it.

- **Item list**: deliberately built the same way as `SettingsWindow`'s token list — a `StackPanel` of manually-constructed rows (`CheckBox` + old-name `TextBlock` + new-name `TextBlock`), *not* `ListBox`/`DataGrid`/`ListView`. **Resizable name columns** (added 2026-07-16, on request — long 유형/패밀리 names were truncated with "..." and couldn't be widened): a header `Grid` above `ItemsPanel` has two named `ColumnDefinition`s (`OldNameColumn`/`NewNameColumn`) with a named `GridSplitter` (`NameColumnSplitter`) between them; since rows are plain `StackPanel`s with no column concept of their own, each row's name/new-name `TextBlock` gets its `Width` copied from the matching column's `ActualWidth` at render time, then kept in sync by `NameColumnSplitter_DragDelta` (fires continuously while dragging) calling `UpdateLayout()` then re-writing every already-rendered row's `Width`. **Two other approaches were tried and live-tested broken first**: a plain `ElementName`+`ActualWidth` `Binding` never updated the rows at all when the splitter moved (only `System.Windows.Data.Binding` compiles — `Binding` alone is ambiguous with `Autodesk.Revit.DB.Binding`, same class of collision as the `Visibility`/`Grid`/`Control`/`Color` gotchas above); `DependencyPropertyDescriptor.FromProperty(ColumnDefinition.ActualWidthProperty, ...)` doesn't even compile in this WPF version (no such public static field) — `ActualWidth` changes on `ColumnDefinition`/`RowDefinition` are a known WPF blind spot for ordinary change-notification mechanisms, hence going straight to the splitter's own `DragDelta` instead.
- **Filter mode combo** (`FilterModeCombo`, added 2026-07-13): the filter textbox alone only ever meant substring-contains; a `ComboBox` next to it (포함됨/포함하지 않음/일치함/일치하지 않음, `SelectedIndex` 0–3) now also supports contains-negated and exact-match(-negated), all case-insensitive via `WorkingNameOf(el)` compared/matched against the filter text in `RenderRows()`. An empty filter textbox always shows everything regardless of mode (including the "않음" modes) — a blank search has no criterion to negate, so treating it as "exclude everything" would be actively wrong. `FilterModeCombo_SelectionChanged` guards on `ItemsPanel == null` for the same reason `ModeRadio_Checked` guards on `ReplacePanel == null`: the ComboBox's XAML-default `SelectedIndex="0"` fires `SelectionChanged` synchronously during `InitializeComponent`, before Row 4's `ItemsPanel` field is connected, so calling `RenderRows()` unguarded there would NRE. **`~로 시작하는`/`~로 끝나는`** (added 2026-07-16, on request): indices 4/5, `string.StartsWith`/`EndsWith` with the same `CurrentCultureIgnoreCase` comparison as the other four modes — added identically to all three filter-mode combos in this codebase (NAMER's `FilterModeCombo`, and 재료 지정's `FilterModeCombo`/`DelFilterModeCombo` in `MaterialAssignWindow`, see below), since all three share the exact same 4-mode `switch` shape that this just extends to 6.
- **Pagination, not virtualization** (root-caused 2026-07-10 after "so slow manual renaming is faster" live feedback): plain `StackPanel` has no UI virtualization, so building a row (4 WPF elements incl. two `TextTrimming`-enabled `TextBlock`s) for *every* matching element up front is the dominant cost — catastrophic for "유형" (every `ElementType` in the doc, easily thousands, no category pre-filter) and compounded further because `FilterBox_TextChanged` used to call the same full `RenderRows()` on *every keystroke*, so typing a filter re-paid that full cost per character. Fixed by capping each render to `PageSize` (200) rows via `RenderMoreRows()` + a "더 보기" button appended after the last row; `RenderRows()` (on category/filter change) now only computes `_filteredElements` (cheap, no UI) and renders the first page, so cost is bounded regardless of category/filter size. `SelectAllButton_Click`/`SelectNoneButton_Click`/`UpdateCountText` operate on `_filteredElements` (the full filtered set), not `_rows` (only the rendered page), so "전체 선택" still reaches items beyond the current page even though they have no on-screen row yet — they'll pick up the correct checked state whenever "더 보기" renders them (`IsChecked = _checkedIds.Contains(el.Id)` at row-creation time).
- **Filtering out an item also unchecks it** (added 2026-07-10, explicit request — "checked" must always mean "currently visible under the filter," never a stale cache from before the filter changed): `RenderRows()` now removes from `_checkedIds` any id that belongs to the *current category* (`_categoryElements`) but isn't in the newly computed `_filteredElements`. Scoped to the current category only, so checked items in other categories (persisted across category switches, per the two-stage-apply design above) are untouched. This runs on every filter keystroke, so narrowing/widening the filter text can silently drop items from the pending selection — expected, not a bug.
- **Drag-to-toggle checkboxes**: each row's `CheckBox` has `IsHitTestVisible=false` and is purely a visual indicator — all click/drag input is handled on the row's `StackPanel` instead (`Background=Brushes.Transparent` set explicitly, since a `null` Background makes a panel's empty space not hit-testable), tagged with its `RenameRow` via `.Tag`. Mouse-down on a row captures the mouse on `ItemsPanel`, computes a single "drag target state" (opposite of the clicked row's current state) once, then every row the cursor passes over during the drag (found via `VisualTreeHelper.HitTest` + walking up to the nearest ancestor with a `RenameRow` tag) gets *set* to that same target state — never toggled relative to its own state, so revisiting a row mid-drag doesn't flicker it back and forth. This is the same reason a plain click still works correctly: it's a drag of length zero, one row set once.
- **Four operations**, computed uniformly by `ComputeNewName(oldName)`: 문자열 치환 (`string.Replace`), 위치에 삽입 (1-based position → `string.Insert` at `position-1`, clamped to name length), 구분자 기준 자리바꾸기 (split at the *first* occurrence of the delimiter, swap the two sides around it — `"ABC-CBA"` with delimiter `"-"` → `"CBA-ABC"`), 구간 삭제 (1-based inclusive start/end position → `string.Remove`, e.g. start=6/end=9 deletes the 6th–9th characters; end clamped to name length, no-op if start is past the end of the name).
- Categories (`View`/`Sheet`/`Family`/`Type`/`Legend`/`Schedule`/`Material`) all collected via `FilteredElementCollector` and treated uniformly as `Element` — `Name` is get/set on `Element` itself (confirmed present and settable on `Element`/`View`/`ViewSheet`/`Family`/`ElementType` via `MetadataLoadContext` reflection before writing any code), so no per-category casting is needed anywhere except the collector queries themselves. `Legend` is *not* a separate Revit class — confirmed via reflection there's no `ViewLegend` type; a legend is a plain `View` with `ViewType == ViewType.Legend`. `Schedule` collects `ViewSchedule` (a `TableView`, itself ultimately a `View`). Since both are `View` subtypes, the generic `View` category explicitly excludes `ViewSchedule` and `ViewType.Legend` (alongside the existing `ViewSheet`/`IsTemplate` exclusions) so nothing appears in two categories at once.
- **`Material` category** (added 2026-07-16, on request): `Material` collects `FilteredElementCollector(doc).OfClass(typeof(Material))` — `Material` is a plain `Element` (not an `ElementType`, so it's absent from the `Type` category's `WhereElementIsElementType()` query), but `Name` is still get/set directly on it, so it slots into the existing uniform-`Element` design with just a new collector case; no changes needed anywhere else (`NamerCommand`'s rename `Transaction` already operates on bare `Element.Name`). `ResolvePreSelectionForCategory` needs no `Material`-specific case either — it falls into the same `default: resolved.Add(el.Id)` branch as View/Sheet/Legend/Schedule, since a pre-selected Material is already the exact element the category list shows (there's no instance→type indirection to resolve, unlike Type/Family).
- **Material rename duplicate-name policy** (added 2026-07-16, on request): renaming a `Material` to a name another `Material` already has would otherwise just fail (Revit requires materials to have unique names) — `NamerWindow` shows a `MaterialDuplicatePanel` (visible only when category is `Material`, radio-toggled, default "병합"; a vertical `StackPanel` — heading line + one `RadioButton` per line, no parenthetical explanation text, per explicit request after the original single-line layout with inline descriptions felt cramped) right under the category radios, and `NamerCommand.RenameMaterial` checks for a same-name `Material` at commit time: if `MergeDuplicateMaterials` is true, `RedirectMaterialUsages` re-scans **every** `ElementType` (not just `MaterialSlotFinder`'s one-slot-per-type view used by the 재료 지정 tool) for **every** compound-structure layer or Material-spec parameter pointing at the old material, repoints each to the existing one, then deletes the now-orphaned old material; if false, `FindUniqueMaterialName` appends `1`, `2`, … (bare digits, per explicit request — not `GetOrCreateSingleLayerWallType`'s `(2)`/`(3)` style) until a free name is found. Scoped to `ElementType`-level usages only — per-instance material overrides are not redirected (known limitation, consistent with `MaterialSlotFinder`'s existing type-only scope). **Confirmed live bug, fixed twice (2026-07-16)**: merge success/failure was originally judged by whether `Document.Delete(material.Id)`'s returned `ICollection<ElementId>` contained `material.Id` — live-tested false-negative (redirect + delete both genuinely succeeded, old material visibly gone, usages correctly moved) yet the returned collection didn't reflect that, so a "병합 실패" warning fired despite total success. First fix (checking `doc.GetElement(material.Id) == null` directly instead of trusting the returned collection) turned out insufficient by itself — live-retested still false-negative immediately after `SetCompoundStructure`/`Parameter.Set` + `Delete` in the same transaction, apparently because Revit hadn't fully regenerated those changes yet at query time. Second fix: call `doc.Regenerate()` right before the `GetElement` check to force those pending changes to fully settle first. If the warning still legitimately fires after this, it likely means the material has a real usage this tool doesn't scan (Object Styles, fill patterns, appearance assets — outside `RedirectMaterialUsages`'s `ElementType`-only scope), not a false positive.
- Pre-selection support mirrors `SplitWallCommand`, with two live-bug fixes on top (2026-07-10, found from an actual bad rename that hit unrelated wire/material types across the whole document — see below):
  - **Instance→type/family resolution** (`ResolvePreSelectionForCategory`): what the user selects in Revit before opening NAMER is almost always model *instances* (a wall, a family instance), not the `ElementType`/`Family` objects themselves — those instance ids never literally match anything in `_categoryElements` for the Type/Family category. Resolve via `element.GetTypeId()` for Type, and `FamilyInstance.Symbol.Family.Id` / `FamilySymbol.Family.Id` for Family, before comparing.
  - **No-match fallback is now "check nothing", not "check everything"**: the old fallback ("if nothing pre-selected matches, check every element in the category") combined with the instance/type mismatch above meant selecting ordinary model elements and switching to "유형" silently checked *every* `ElementType` in the entire document — including completely unrelated disciplines (electrical wire/conductor types, materials) — which then got renamed for real on "최종 적용". Confirmed live via a screenshot showing `Steel`/`Copper`/`THHN`/`RHW-2`-style wire-type names mangled by an unrelated insert-at-position-5 operation. This default-check logic only runs the *first* time each category is loaded in a given window session (`_categoriesInitialized`).
- **`NamerCommand`'s rename `Transaction` deliberately sets NO custom `IFailuresPreprocessor`** — it originally had one (`SilentWarningsPreprocessor`, the same `DeleteWarning`/`ResolveFailure` pattern as `SplitWallCommand.SilentFailuresPreprocessor`, added to suppress unrelated pre-existing warnings surfaced by the full-document regenerate that any `Transaction.Commit()` triggers), but this caused a **confirmed live bug (2026-07-10): renames silently never reached the model** — no error dialog, Revit's status-bar progress gauge ran normally, but the wall/type names were unchanged after commit. Root-caused by checking `Transaction.Commit()`'s return value (previously ignored): it was returning `TransactionStatus.RolledBack` with zero exceptions thrown. Diagnostic instrumentation (counting `PreprocessFailures` calls) showed it fired **exactly 666 times with zero failure messages every time**, identically whether 100 items or only 2–3 items were checked — proving the loop was unrelated to what was being renamed. Removing the custom preprocessor entirely (commenting it out, then deleting it) fixed the rollback completely; live-confirmed the rename commits normally with no preprocessor at all. Root Revit-engine mechanism still unconfirmed, but merely *attaching* any custom `IFailuresPreprocessor` to this specific transaction is what breaks it — **do not re-add one here**, even the "safe" delete-warnings/resolve-errors pattern that works fine in `SplitWallCommand`. Since this transaction only renames elements (no geometry edits), unresolved warnings from unrelated pre-existing issues are rare in practice, so losing the suppression isn't a major loss. `Transaction.Commit()`'s `TransactionStatus` return value is still checked and surfaced via a `TaskDialog` if ever non-`Committed`, so a future rollback (whatever the cause) won't fail silently again.
- Renames happen in one `Transaction`, per-item `try/catch` (duplicate names, locked elements, etc. are common failure causes) with a summary `TaskDialog` listing failures — the same "collect skip reasons, one summary dialog" pattern as `SplitWallCommand`.
- **Two-stage apply** (added 2026-07-10, on request — chaining several operations without reopening NAMER each time was too tedious): "적용" only mutates two in-memory dictionaries, `_trueOriginalNames` (name when NAMER first saw that element this session) and `_workingNames` (accumulated result of every "적용" click so far) — it never touches the Revit model and the window stays open, so the user can switch category/mode and keep layering more operations on top of the same working names. "최종 적용" is the only button that sets `Result` and closes the dialog — it diffs `_workingNames` against `_trueOriginalNames` across *all* categories touched this session, not just the currently-visible one. "취소" discards everything for free, since nothing was ever written to Revit until this point — the whole in-window editing session is implicitly undoable by just not clicking "최종 적용". **Both dictionaries are seeded lazily, only for elements actually renamed** (`ApplyButton_Click` writes `_trueOriginalNames[el.Id]` the first time that specific element's name changes, never upfront) — an earlier version seeded every element in the *entire* loaded category (checked or not) as soon as `LoadCategory` ran, which meant opening "유형" alone did a `Name` read across every `ElementType` in the document before the user touched anything; flagged live as a real contributor to NAMER feeling sluggish and removed. `WorkingNameOf` already falls back to a live `el.Name` read when there's no dictionary entry, so nothing else needed to change for elements that stay untouched all session.
- **Unchecked items never get a *new* computed change, but a change already committed via "적용" survives being unchecked afterward** — this is a reversal (2026-07-13) of the original hard rule ("always gate the final result on current `_checkedIds`"). `ApplyButton_Click` still only computes `ComputeNewName` for currently-checked elements (so an item that was never checked can never enter `_workingNames` — that half of the original rule is untouched). But `FinalApplyButton_Click`/`UpdatePendingChangesText` **no longer** filter `_workingNames` by `_checkedIds.Contains(...)` — they just diff `_workingNames` against `_trueOriginalNames` directly. Root cause: `RenderRows()` auto-unchecks items that scroll out of the current filter (see "Filtering out an item also unchecks it" above), and with the old checked-gate this **silently and permanently discarded already-applied changes** the moment the user typed a different filter — live-reported as "필터에서 사라지면서 최종적용을 할 수 없게 됨". Since a change can only ever enter `_workingNames` through a deliberate checked "적용" click, letting it survive a later filter-driven auto-uncheck is safe and matches user intent; only an actually-untouched (never-applied) item is still excluded. `UpdateRowPreview` is unaffected by this change — unchecked rows still show their already-applied working name in gray without recomputing anything.
- **`ApplyButton_Click` clears every mode's input textboxes** (`ClearModeInputs`) right after a successful in-window apply — found live: leftover values (e.g. a still-filled 구간 삭제 start/end) would otherwise sit in the boxes and get re-applied on a second stray "적용" click, silently deleting another range from the just-shortened name. Safe to blanket-clear because every mode already treats empty/unparseable input as a no-op in `ComputeNewName`. `FinalApplyButton_Click` structurally never calls `ComputeNewName` at all — it only ever moves already-`_workingNames`-accumulated strings into `Result` — and neither apply button carries `IsDefault` (removed on request, "혹시 모르니") specifically so a stray Enter keypress while typing in a param box can't trigger either apply path unintentionally.
- Gotcha hit while writing this: an unqualified `Visibility.Visible`/`Visibility.Collapsed` inside a `Window`-derived class's code-behind is a compile error (`CS0176`), *not* what you'd expect — `FrameworkElement` (which `Window` inherits) has an instance property literally named `Visibility`, so the bare identifier resolves to `this.Visibility` instead of the `System.Windows.Visibility` enum; must fully qualify as `System.Windows.Visibility.Visible`. Same category of gotcha as the `Grid`/`Control`/`Color` collisions above — always fully-qualify when a WPF type name might collide with either an Autodesk.Revit.DB type or, as here, an inherited instance member.

## 재료 지정 (Material Assign)

A batch material-assignment tool, added 2026-07-16 on request ("여러 유형을 한번에 선택해서 재료를 지정"), own ribbon panel/button ("재료 지정"→`MaterialAssignCommand`). `MaterialAssignWindow` deliberately reuses NAMER's proven UI skeleton wholesale (filter textbox + 포함/일치 mode combo, checkbox-row `StackPanel` with drag-to-toggle, pagination, two-stage 적용/최종 적용 with the same "checked-gate only applies to *new* changes, not to changes already committed via 적용" rule) — the only real difference from NAMER is *what* gets changed: an `ElementId` (a `Material`) instead of a `string` (a name). **`FilterTargetCombo`** (added 2026-07-16, on request): a 패밀리/유형-vs-현재 재료 selector next to the filter textbox — `RenderRows()` matches the filter text against `c.Type.Name` or `CurrentMaterialName(c)` depending on its `SelectedIndex`, so a user can search by "which types currently use material X" instead of only by type name; guarded the same `ItemsPanel == null` way as `FilterModeCombo`. **"재료 없음" filter target** (added 2026-07-16, on request — third `FilterTargetCombo` item, index 2): shows only candidates whose `CurrentMaterialId(c) == ElementId.InvalidElementId`, bypassing `FilterBox`/`FilterModeCombo` entirely (a boolean condition has no text to search) — both are explicitly `IsEnabled = false` while this mode is selected, since a filter box/mode a user could still type into but that does nothing would be confusing. **Unused-material labeling** (added 2026-07-16, on request — "패밀리/유형이 지정 안 된 재료도 알아볼 수 있었으면"): `LoadCandidates()` now also builds `_usedMaterialIds` (every `MaterialId` any candidate's slot actually references), and `LoadMaterials()` (which must run *after* `LoadCandidates()` for this) wraps each `Material` in a `MaterialOption{ DisplayName }` appending "(지정된 유형 없음)" for materials no type currently references — shown both in the assign tab's `MaterialCombo` (`DisplayMemberPath="DisplayName"`) and the delete tab's rows (`MaterialDisplayName(m)`), so an orphaned material is now visible and pickable everywhere instead of silently invisible in a tool whose candidate list is fundamentally type-centric. **Confirmed live bug, fixed (2026-07-16)**: `MaterialOption.Material`/`DisplayName` were originally plain fields — `DisplayMemberPath` only binds to *properties* (its reflection lookup is `PropertyDescriptor`-based and simply doesn't see fields), so every row in `MaterialCombo` silently rendered as blank text even though `ItemsSource` genuinely held all the materials — looked exactly like "no materials show up" despite the combo being fully populated. Fixed by making both `{ get; set; }` properties. **Stale-checked-cache fix** (added 2026-07-16, confirmed live — "체크했던 게 캐시처럼 계속 남아있으면 안 된다"): `ApplyButton_Click` used to leave applied rows checked, so picking a *different* material and clicking 적용 again silently re-applied it to those still-checked rows too; it now removes each successfully-applied row's id from `_checkedIds` right after applying.
- **`MaterialSlotFinder`** (shared by the window's display and `MaterialAssignCommand`'s commit): finds the single "current material" slot on an `ElementType`, if any. For `HostObjAttributes` types (Wall/Floor/Roof/Ceiling — anything with `GetCompoundStructure()`), only types with **exactly one non-membrane layer** qualify (ambiguous which layer to change otherwise) — this is exactly the shape of the single-layer types `SplitWallCommand`/`SplitFloorCommand` themselves produce, so it's this tool's main intended target. Other types (doors/windows/furniture/framing, etc.) qualify if they have a `Parameter` whose `Definition.GetDataType() == SpecTypeId.Reference.Material`; the first such parameter found is used. Types matching neither are simply absent from the list (no read-only "N/A" rows).
- `MaterialSlotFinder.Apply` re-runs `Find` fresh at commit time inside the transaction rather than reusing anything computed while the window was open, and for the compound-structure case rebuilds the whole layer list via `SetLayers` (a `CompoundStructureLayer` is immutable — there is no per-layer material setter — so the target index's layer must be replaced wholesale with a new one carrying the same width/function). **Resizable columns** (added 2026-07-16, on request): the "재료 지정" tab got the same header-`Grid`-with-named-`ColumnDefinition`s-and-`GridSplitter`s treatment as NAMER (see NAMER's "Resizable name columns" note above for why plain `Binding`/`DependencyPropertyDescriptor` don't work) — `TypeNameColumn`/`CurrentMaterialColumn`/`NewMaterialColumn`, two splitters, one shared `ColumnSplitter_DragDelta` handler that `UpdateLayout()`s then rewrites all three `TextBlock` widths on every already-rendered row.
- `MaterialAssignCommand`'s transaction also deliberately sets **no custom `IFailuresPreprocessor`**, for the same precautionary reason as `NamerCommand` (see its confirmed-live-bug note above) even though this hasn't itself been live-confirmed to break here — untested risk, not worth taking for a rarely-warning-triggering metadata edit. **Post-write verification** (added 2026-07-16, on request — "정확하게 모델에 적용이 되었으면 좋겠다"): neither `MaterialSlotFinder.Apply`'s `bool` return nor `Document.Delete`'s lack of a thrown exception is trusted as proof the change actually landed — `ExecuteAssign` re-runs `MaterialSlotFinder.Find` right after applying and compares its `MaterialId` against the intended new material, and `ExecuteDelete` checks that `Document.Delete`'s returned `ICollection<ElementId>` actually contains the target id; either mismatch is now reported in the same failure-summary `TaskDialog` instead of being silently treated as success.
- **"재료 삭제" tab** (added 2026-07-16, on request): a second `TabItem` in the same `MaterialAssignWindow`, independent of the "재료 지정" tab's type list — lists every `Material` in the doc directly (not filtered by any type relationship) with its own filter+mode combo and drag-checkable rows (`DelFilterBox`/`DelItemsPanel`/`DeleteRow`, same copy-pasted drag-select mechanics as the assign tab and `NamerWindow`, since this codebase prefers duplicating the small UI pattern over sharing it across unrelated row types). No pagination here (unlike the assign tab's `PageSize`/"더 보기") since a document's `Material` count is normally far smaller than its `ElementType` count. Deletion has no "computed new value" to preview, so there's no two-stage 적용/최종 적용 — checking rows and clicking "선택한 재료 삭제" goes straight to a Yes/No `MessageBox` confirmation, then closes the dialog with `DeleteResult` set; the actual `doc.Delete(materialId)` calls still happen only afterward, inside `MaterialAssignCommand`'s own transaction (`ExecuteDelete`), same as every other command in this project — the window itself never touches the model. Since a `ShowDialog()` session can only produce **one** outcome, `MaterialAssignCommand.Execute` checks `window.DeleteResult` before `window.Result`/`window.IdentityResult` and dispatches to `ExecuteAssign`/`ExecuteDelete`/`ExecuteIdentityEdit` accordingly (mutually exclusive per invocation — do multiple kinds of operations by reopening the tool again for each).
- **"클래스/설명 변경" tab** (added 2026-07-16, on request — "재료의 아이덴티티의 클래스 및 설명을 변경"): a third `TabItem`, same NAMER-derived filter+mode combo (identical 6-mode `FilterModeCombo` semantics, `IdFilterBox`/`IdFilterModeCombo`) searching `_allMaterials` by name (no pagination, same reasoning as the 재료 삭제 tab). A `IdTargetFieldCombo` (클래스/설명) + one shared `IdNewValueBox` textbox, plus the same 2-stage 적용/최종 적용 (`_trueOriginalClass`/`_workingClass` and `_trueOriginalDescription`/`_workingDescription`, two independent dictionary pairs so a given 적용 click only ever touches whichever one field was targeted) as the 재료 지정 tab. Each row shows both 클래스 and 설명 columns (`IdColumnSplitter_DragDelta`-resizable, same pattern as the other tabs) — whichever column matches the *currently selected* target field, for currently-checked rows with a differing new value, previews as `current → new` in black; everything else shows the accumulated working value in gray, live-updated on every `IdTargetFieldCombo`/`IdNewValueBox` change via `IdRefreshPreview()`. `IdFinalApplyButton_Click` diffs both dictionaries independently and only emits a `(MaterialId, string? NewClass, string? NewDescription)` tuple when at least one field actually changed from its true original — a single material can have only its class touched, only its description, or both, across separate 적용 clicks in one session. `MaterialAssignCommand.ExecuteIdentityEdit` writes `Material.MaterialClass` directly (a real settable property, confirmed via `MetadataLoadContext` reflection of the 2026 `RevitAPI.dll` the same way `Floor.SlabShapeEditor` was checked) but writes 설명 via `Material.get_Parameter(BuiltInParameter.ALL_MODEL_DESCRIPTION)` — **this parameter binding is the best-known candidate (the same general-purpose "Description" field used across many other Identity Data tabs) but is not itself live-confirmed against a real Material's Identity Data tab in Revit; verify live before trusting it**, since reflection can prove a property/enum member exists but not which parameter a given element type actually exposes it as. Same "no custom `IFailuresPreprocessor`, verify by re-reading after Set(), report status/failed-list via `TaskDialog`" pattern as `ExecuteAssign`/`ExecuteDelete`.

## Multi-version build (Revit 2023–2027)

`WallSplitter.csproj` builds the **same source** against 5 different Revit API versions via custom per-year Configurations, not `<TargetFrameworks>` (a single project can't multi-target the same TFM twice, and 2025/2026 share `net8.0-windows` while still needing different `RevitAPI.dll` versions):

- Configurations are named `Debug2023`/`Release2023` … `Debug2027`/`Release2027`. `WallSplitter.csproj` reads the trailing 4 digits into an MSBuild `$(RevitVersion)` property, which drives both `$(TargetFramework)` and the `Nice3point.Revit.Api.RevitAPI`/`RevitAPIUI` NuGet package version (`Version="$(RevitVersion).*-*"`).
- **Empirically confirmed** (by restoring the Nice3point packages and diffing reflected API surface across the 2023 and 2027 packages — not guessed): 2023 & 2024 → `net48`; 2025 & 2026 → `net8.0-windows`; 2027 → `net10.0-windows`. If Autodesk ships a Revit 2028, check its actual required TFM the same way (restore `Nice3point.Revit.Api.RevitAPI` at `2028.*-*` against candidate TFMs and read the `NU1202`/no-warning result) before assuming it follows the same pattern.
- All Wall/Sketch/Curve/Transform/Plane/Ribbon APIs this project uses were verified identical across the 2023 and 2027 packages, so most of the source is shared as-is across all 5 years with no conditional compilation. One exception exists (`Floor.SlabShapeEditor`, see `SplitFloorCommand.cs` above): when a future API call turns out to differ by year, gate it with `#if REVIT20XX`/`#if REVIT20XX_OR_GREATER`-style constants defined in a `DefineConstants` `PropertyGroup` keyed off `$(RevitVersion)` (Nice3point's SDK generates these automatically; since we don't use their SDK, add each constant by hand, only for the specific years actually needed — don't pre-define the full matrix speculatively).
- We use the **`Nice3point.Revit.Api.RevitAPI`/`RevitAPIUI` NuGet packages** (not local `Program Files\Autodesk\Revit <year>\` HintPath references) specifically so the project builds for all 5 years even though this machine only has Revit 2023 and 2026 actually installed. These are `ref/`-only (compile-time) packages and are never copied to output.
- We deliberately did **not** adopt `Nice3point.Revit.Sdk` (the custom MSBuild SDK that automates this exact Configuration→TFM→package wiring, used by their official templates) — the mapping above is hand-rolled in plain `Microsoft.NET.Sdk` `PropertyGroup` conditions to keep the build fully transparent/inspectable, consistent with how the rest of this repo avoids extra abstraction. Revisit that decision if maintaining the per-year conditionals by hand becomes painful.
- `<LangVersion>latest</LangVersion>` is required — `net48` alone defaults to C# 7.3, which breaks `Nullable`/target-typed patterns used in the shared source.
- Build one year: `dotnet build WallSplitter/WallSplitter.csproj -c Release2026`. Output lands in `WallSplitter/bin/Release2026/net8.0-windows/WallSplitter.dll` (folder name includes the year since Configuration is part of the default output path).

## Project structure

- `WallSplitter.slnx` — solution file (slnx format), references two projects:
  - `WallSplitter/WallSplitter.csproj` — the add-in itself.
  - `SunnyToolsInstaller/SunnyToolsInstaller.csproj` — a standalone installer console app (see below).
- `WallSplitter/WallSplitter.addin` — **dev-mode** manifest whose `<Assembly>` path points directly at `WallSplitter/bin/Debug2026/net8.0-windows/WallSplitter.dll` (this machine's primary Revit version, 2026). This file (or a copy of it) is what's registered in `%APPDATA%\Autodesk\Revit\Addins\2026\` on this machine today for day-to-day dev testing. Do not treat it as the deployment mechanism for other users/versions — it's a debug convenience for one specific year.

## Installer

`SunnyToolsInstaller/` is a self-contained console app that installs **all 5 year-builds** of `WallSplitter.dll` for end users who don't have this repo, auto-detecting which Revit version(s) they actually have. It's named/branded "Sunny Tools" (not "WallSplitter") since it installs the whole ribbon tab, not just the wall-split command — but the underlying add-in project, assembly, namespace, and Revit `.addin` `<Name>`/`FullClassName` are still `WallSplitter` internally (not renamed; only the installer's own project/output/branding changed on 2026-07-10).

- It does **not** use `ProjectReference` to `WallSplitter.csproj` at all (not even with `ReferenceOutputAssembly="false"`) — instead, an MSBuild `Target BeforeTargets="Build;Publish"` shells out `dotnet build ...WallSplitter.csproj -c Release202X` five times (once per year), then `<None Include="...\net48\*.dll" TargetPath="Payload\202X\%(Filename)%(Extension)">`-style wildcard items copy **every DLL** in each year's output folder (not just `WallSplitter.dll`) into a `Payload\<year>\` subfolder of the installer's own output. This keeps the installer's own dependency graph (and therefore its self-contained publish size) completely unrelated to `WallSplitter.csproj`'s `UseWPF=true` — a plain `ProjectReference` would drag the WPF native runtime into the installer and balloon it from ~30MB to ~65MB zipped for no reason, since the installer process itself never loads WPF, it only copies DLLs as bytes.
  - **Why every DLL, not just `WallSplitter.dll`**: net48 (2023/2024) doesn't ship `System.Text.Json` in the framework, so `NamingSettings`' JSON persistence pulls in `System.Text.Json.dll` + several polyfill DLLs (`System.Memory`, `System.Buffers`, etc.) that NuGet copies alongside `WallSplitter.dll` in the net48 build output. If the installer only copied the one DLL, WallSplitter would fail to load in Revit 2023/2024 with a missing-assembly error. net8.0-windows/net10.0-windows (2025–2027) have `System.Text.Json` in-box, so their payload is still just the one DLL — the wildcard handles both cases uniformly without needing to special-case per year.
- At runtime (`Program.cs`) it checks `C:\Program Files\Autodesk\Revit <year>\Revit.exe` for each of 2023–2027 to detect which versions are actually installed, then for each detected year copies **every file** in `Payload\<year>\` into `%APPDATA%\Autodesk\Revit\Addins\<year>\WallSplitter\` and writes a matching `.addin` manifest there (same `AddInId` as the dev manifest). If detection finds **zero** installed versions (e.g. non-default install path), it falls back to installing all 5 years defensively rather than silently doing nothing.
- `RuntimeIdentifier=win-x64`, `SelfContained=true`, and `PublishSingleFile=true` are set directly in `SunnyToolsInstaller.csproj` (not passed as `-r`/`--self-contained` flags) so the published output is just `SunnyTools_Install.exe` (+ `.pdb`) instead of the exe sitting among dozens of loose .NET runtime DLLs — added 2026-07-10 after that was reported as hard to find. One consequence: `PublishSingleFile`'s bundler only bundles managed assemblies, not the `Payload\` content folder, and leaves it behind in the intermediate `bin\...\` output — a `Target Name="CopyPayloadToPublishDir" AfterTargets="Publish"` re-copies `$(OutDir)Payload` into `$(PublishDir)Payload` to compensate; don't remove that target or Payload will silently go missing from the publish folder again.
- Distribute via: `dotnet publish SunnyToolsInstaller/SunnyToolsInstaller.csproj -c Release -o <fresh-output-dir>` (RID/self-contained/single-file all come from the csproj now, no need to pass them) → zip that folder's contents (still needs the sibling `Payload\` folder, just no longer needs the exe's old DLL siblings) and send it; recipient extracts the zip and double-clicks `SunnyTools_Install.exe`. A pre-built copy is kept at `SunnyTools_Installer.zip` in the repo root.
- This machine's Desktop folder appears to be watched by some kind of ransomware-canary/decoy-file protection (Windows Controlled Folder Access or similar): publishing ~190+ files in one burst here can transiently spawn an undeletable `*.DOCX` file with garbage attributes in the output folder, which then disappears on its own. If `Remove-Item` on a `bin`/`publish` folder fails with "Invalid File or Directory attributes value", don't fight it — publish to a fresh `-o` directory instead (e.g. `..._v2`, `..._v3`) and ignore the stale one; if it also blocks a wildcard `Copy-Item`/`Get-ChildItem *`, copy files by explicit name instead (the phantom entry breaks glob expansion but not a direct named path).
- **CONFIRMED STALE-PAYLOAD BUG (2026-07-10)**: the `<None Include="..\WallSplitter\bin\Release202X\...\*.dll" CopyToOutputDirectory="PreserveNewest">` items that stage `Payload\<year>\` do **not** reliably pick up a fresh `WallSplitter.csproj` rebuild — confirmed live: after fixing a bug in shared source and rebuilding all 5 `Release202X` configs (verified fresh via direct file timestamps), `SunnyToolsInstaller`'s own `bin\Release\...\Payload\<year>\WallSplitter.dll` still held the pre-fix bytes/timestamp; a fresh `dotnet publish -o` output inherited that same stale copy. Root cause not fully isolated (suspected MSBuild incremental up-to-date caching on the wildcard item, since `bin`/`obj` couldn't be deleted to force a clean rebuild — see the ghost-file issue above), but the fix that worked was to manually copy the real `WallSplitter\bin\Release202X\<tfm>\*` files over the publish output's `Payload\<year>\` by explicit filename, post-publish. **Do not trust `SunnyTools_Installer.zip` (or any fresh publish) to reflect a just-made source fix without verifying `Payload\<year>\WallSplitter.dll`'s timestamp/size against the corresponding `WallSplitter\bin\Release<year>\...\WallSplitter.dll` directly** — silent staleness here means end users (or this machine's own installed add-in) keep running old, already-"fixed" code with zero error signal.
